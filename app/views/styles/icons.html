Notes

NHS Choices and wider NHS products have amassed a number of different icons.

Icon's can be hard for users to understand; there are very few icons that universally recognised - in many cases its almost like learning a new language for some users.

Rather than recreating what was there before, we needed to assess whether user needs existed for icons.

Design approach

If there was a distinct user need for icons we felt it would be good that the icons sit together in a set (or family).

As the NHS website is striving to be more personal we felt that some iconography can be quite cold and clinical. We made a conscious decision to round the corners of our icons to subtly soften their presence.

Colours of icons are governed by context (where it sits on the page, what colour background etc). Icons also need to meet accessible colour contrast standards (recommended tool to test colour combinations: https://webaim.org/resources/contrastchecker/)

Icon sizes

When creating an icon, view and edit it at 24px x 24px. By maintaining this ratio, any changes to the original will be scaled up or down proportionally, which preserves sharp edges and correct alignment.

Example: Close icon on 24px x 24px grid

Our icons are drawn at this scale using a stroke weight of 2px and rounded caps and joins.

To prevent pixel distortion (and to get sharp edges), we ensure our icons fit to the pixel grid.

Final icon sizes are dictated by the context - for example the cross icon to close the navigation menu on mobile is larger then the chevron used to separate breadcrumb links.

Sizes for icons are governed by the following

  1. If they intended to be clicked (actions)
  2. If they're sat within a button
  3. What they're intended to be clicked with (mouse or finger)
  4. If they're not actionable

If the icon is an intended touch target, we try and meet Google's recommended accessible target size of 48px where possible (https://developers.google.com/web/fundamentals/accessibility/accessible-styles).

User need

As a user I need to easily identify, understand and navigate content, so that I get the best outcome for my situation.

Whats in alpha / beta / live

A small number icons were being used in beta, the only ones of note being action link arrows https://beta.nhs.uk/

Research findings

We tested the icons in context, not on their own - so they appeared within components on a full page.

Research findings relate to the success of the full component on the finished page.

Do and Don't (tick and cross)

Do and Don’t boxes positives: users liked the ticks and crosses and the colour schemes associated with each, with the green reinforcing the ‘do’ positive action and the red being a warning and don’t colour.
(User research lab 6 participants 25th May 2018 London and popup 22 participants 25th May 2018 Leeds, The Light shopping centre)

Action link (arrow)

Most users understood the purpose of the action-link, but some felt unsure what to expect when clicking on it.
(usability lab Salford, 4th July 2018)

Reveal/expander (plus and minus)

All users tapped the reveal, which behaved as they expected
(usability lab Salford, 4th July 2018)

Pagination (arrows)

One user found the pagination particularly useful for navigating based on the way he read content and it supporting his mental model of the page structure.
(usability lab Salford, 4th July 2018)

4/5 users – including a user with very low digital skills – used pagination to navigate through pages, and knew this was associated with the navigation at the top of the page.
(usability lab, Sheffield, 16th July 2018)

Our rules for using icons are

  • Only use icons when there’s a real user need for them (avoid unnecessary decoration)
  • Use icons sparingly - most people won’t understand an icon vocabulary
  • If used, ensure icons are clear, simple and accompanied by relevant text
  • Don’t hide functionality under icons

User need

As a user I need to easily identify, understand and navigate content, so that I get the best outcome for my situation and don't endanger my health.

How to use icons

We inline SVG icons for performance and to have the ability to control the appearance of icons with CSS. This decision was inspired by CSS Tricks - A Pretty Good SVG Icon System and CSS Tricks - A Complete Guide to SVG Fallbacks.

Using a server side or templating language you can include the icons with include 'partials/icons/icon-search.svg'.

SVG icons have the class .nhsuk-icon which set a default size for the icon and each icon has a specific class such as .nhsuk-icon__search which can modify specific icons with styles such as fill. This means you can change the colour of the SVG for states such as hover, focus and active.

Each SVG icon has a PNG fallback which means we can support browsers such as Internet Explorer 8. The fallback images have a class such as .nhsuk-icon__search--fallback which can be used to modify sizing and spacing of the PNG fallbacks.

SVG icons and logos must be accessible, read more about accessibile SVGs on CSS Tricks website.
