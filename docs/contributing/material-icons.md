# Material icons workflow

This guide explains how Material icons are stored, built, consumed, and documented in this repository.

## Who this is for

- Engineers contributing to the toolkit icon set
- Engineers consuming toolkit icons in Nunjucks/templates
- Engineers maintaining the documentation site

## At a glance

1. Source files and metadata live in [`packages/toolkit/assets/icons/material/`](../../packages/toolkit/assets/icons/material/).
2. A build script converts those SVGs into one sprite file: [`packages/toolkit/assets/icons/material-sprite.svg`](../../packages/toolkit/assets/icons/material-sprite.svg).
3. Toolkit components render icons by name via the icon macro and `<use href="...#ofh-icon-name">`.
4. The docs site reads the same manifest and renders the icon gallery from it.

## Where the icon assets live

- [`packages/toolkit/assets/icons/material/manifest.json`](../../packages/toolkit/assets/icons/material/manifest.json): source of truth for icon names, categories, and default sizes.
- [`packages/toolkit/assets/icons/material/*.svg`](../../packages/toolkit/assets/icons/material/): one SVG file per icon.
- [`packages/toolkit/assets/icons/material-sprite.svg`](../../packages/toolkit/assets/icons/material-sprite.svg): generated sprite used at runtime.

## How sprite generation works

The sprite is generated by [`packages/toolkit/scripts/build-material-icon-sprite.js`](../../packages/toolkit/scripts/build-material-icon-sprite.js):

- Reads all SVG files from `assets/icons/material`
- Sorts them by filename for deterministic output
- Converts each file into a `<symbol id="ofh-icon-<name>">`
- Writes `assets/icons/material-sprite.svg`

Run it directly from repo root:

```bash
pnpm --filter @ourfuturehealth/toolkit exec node scripts/build-material-icon-sprite.js
```

It also runs automatically in toolkit build via gulp:

- [`packages/toolkit/gulpfile.js`](../../packages/toolkit/gulpfile.js) `buildMaterialIconSprite`
- `gulp build` task sequence

### Why use a sprite instead of inlining SVG paths everywhere

- One shared source for icon path data
- Consistent API (`name` + `size`) across components
- Smaller repeated markup in templates (`<use>` instead of full path content each time)
- Easy to update icon internals without touching every component template

## How icons are used in toolkit code

Toolkit components use the shared icon macro:

- [`packages/toolkit/components/icon/macro.njk`](../../packages/toolkit/components/icon/macro.njk)

The macro renders:

- An `<svg>` element with `.ofh-icon` classes
- A `<use href="/assets/icons/material-sprite.svg#ofh-icon-<name>">`

Example usage:

```njk
{% from 'icon/macro.njk' import icon %}

{{ icon({ "name": "search", "size": 24, "title": "Search" }) }}
```

Core icon styles are in [`packages/toolkit/core/styles/_icons-material.scss`](../../packages/toolkit/core/styles/_icons-material.scss) and imported in [`packages/toolkit/core/all.scss`](../../packages/toolkit/core/all.scss).

## React components status and future section

Current status: the React package does not yet have a shared icon component.

When React icon support is added, this doc should be updated with the final approach. Recommended baseline:

- Keep icon names aligned with toolkit `manifest.json`
- Reuse the same icon semantics (`name`, `size`, decorative vs titled)
- Decide implementation explicitly:
  - Sprite-based `<use>` approach (consistency with toolkit/docs), or
  - Build-time inlined SVG components

Related package: [`packages/react-components/`](../../packages/react-components/)

## How icons are shown in the documentation site

The docs site does not hardcode the icon list. It reads toolkit manifest data:

- Data source: [`packages/site/views/_data/materialIcons.js`](../../packages/site/views/_data/materialIcons.js)
- Gallery page: [`packages/site/views/design-system/styles/icons/index.njk`](../../packages/site/views/design-system/styles/icons/index.njk)

Flow:

1. `materialIcons.js` loads `toolkit/assets/icons/material/manifest.json`
2. It sorts and groups icons by category
3. The icons page loops through those groups and calls the shared icon macro

The sprite is copied into docs output by Eleventy:

- [`packages/site/eleventy.config.js`](../../packages/site/eleventy.config.js)

## Add new icons from Figma

Use this checklist whenever Design/Figma introduces new icons.

1. Confirm approval.
   Ensure the icon is approved for shipping and aligned with [`do-not-track/iconography-updates.md`](../../do-not-track/iconography-updates.md).

2. Define stable metadata.
   Choose a stable kebab-case name (for example `appointment-booking`), select a category from `Data Validation`, `Action`, `Arrows`, `Graphical`, `Stepper`, `Socials`, and choose a `defaultSize` of `16`, `24`, or `32`.

3. Export and add the SVG.
   Export SVG from Figma, add it as `packages/toolkit/assets/icons/material/<name>.svg`, and use `currentColor` where colour should follow CSS.

4. Update manifest metadata.
   Add an icon entry in [`manifest.json`](../../packages/toolkit/assets/icons/material/manifest.json) with `name`, `category`, and `defaultSize`.

5. Rebuild sprite

```bash
pnpm --filter @ourfuturehealth/toolkit exec node scripts/build-material-icon-sprite.js
```

6. Run checks

```bash
pnpm --filter @ourfuturehealth/toolkit build
pnpm --filter @ourfuturehealth/toolkit lint
```

7. Verify the docs site.
   Run docs locally (`pnpm dev:site`) and confirm the icon appears in `/design-system/styles/icons` in the expected category and sizes.

8. Use the icon in components.
   Render it with the icon macro using the new `name`, then add or update component docs where relevant.

## Common pitfalls

- The sprite builder includes all `*.svg` files in the folder. The docs gallery uses `manifest.json`.
  - If an SVG exists but manifest entry is missing, it may be in sprite but not docs gallery.
- New categories will not appear automatically unless added to category order in [`packages/site/views/_data/materialIcons.js`](../../packages/site/views/_data/materialIcons.js).
- Avoid using [`packages/toolkit/scripts/generate-material-icons.js`](../../packages/toolkit/scripts/generate-material-icons.js) for normal Figma imports; it can overwrite SVG files with hardcoded/placeholder content.
