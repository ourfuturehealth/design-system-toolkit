# Release Versioning Strategy

## Overview

This document explains the versioning strategy for releases after the v4.0.0 monorepo restructure.
For the operational step-by-step release checklist, see [Release Process](./release-process.md).

## The Change: Single Package → Monorepo

### Before (v3.4.2 and earlier)

- **One package**: The entire repository was the toolkit
- **One version**: Tags like `v3.4.2` represented the toolkit version
- **Release assets**: Toolkit bundle zip + source code
- **Installation**: `"ofh-design-system-toolkit": "github:ourfuturehealth/design-system-toolkit#v3.4.2"`

### After (v4.0.0 and later)

- **Multiple packages**: Monorepo with `toolkit` and `react-components`
- **Independent versioning**: Each package has its own version
- **Release strategy**: Releases are per-package, not per-monorepo
- **Installation**: Must specify package subdirectory

## Release Tag Patterns

### Toolkit Releases

**Pattern:** `toolkit-v*` (e.g., `toolkit-v4.0.0`, `toolkit-v4.0.1`, `toolkit-v4.1.0`)

**Why:** Clear identification that this is a toolkit release in a multi-package repository.

```bash
# Create toolkit release
git tag toolkit-v4.0.0
git push origin toolkit-v4.0.0
```

**Installation:**

```json
{
  "dependencies": {
    "@ourfuturehealth/toolkit": "github:ourfuturehealth/design-system-toolkit#toolkit-v4.0.0:packages/toolkit"
  }
}
```

**Release contains:**

- Toolkit bundle zip: `ofh-design-system-toolkit-4.0.0.zip`
- Source code (auto-generated by GitHub)

### React Components Releases

**Pattern:** `react-v*` (e.g., `react-v0.0.1`, `react-v0.0.2`, `react-v0.1.0`)

```bash
# Create react components release
git tag react-v0.0.1
git push origin react-v0.0.1
```

**Installation:**

```json
{
  "dependencies": {
    "@ourfuturehealth/react-components": "github:ourfuturehealth/design-system-toolkit#react-v0.0.1:packages/react-components"
  }
}
```

**Release contains:**

- Source code (auto-generated by GitHub)
- Built React components (via prepare script)

### Combined Releases (Optional)

If both packages are updated together, you can create both tags:

```bash
git tag toolkit-v4.1.0
git tag react-v0.1.0
git push origin toolkit-v4.1.0 react-v0.1.0
```

This creates two separate releases.

## What This Means for Existing Projects

### If You're Using v3.4.2 or Earlier (Pre-Refactor)

**Current installation:**

```json
{
  "dependencies": {
    "ofh-design-system-toolkit": "github:ourfuturehealth/design-system-toolkit#v3.4.2"
  }
}
```

**✅ This continues to work - no action required!**

Git tags are immutable, so v3.4.2 will always point to the pre-refactor code. You can continue using it indefinitely.

**To upgrade to v4.0.0+ (Post-Refactor):**

Only when you're ready to upgrade, change to the new syntax:

```json
{
  "dependencies": {
    "@ourfuturehealth/toolkit": "github:ourfuturehealth/design-system-toolkit#toolkit-v4.0.0:packages/toolkit"
  }
}
```

**Changes required:**

1. Package name: `ofh-design-system-toolkit` → `@ourfuturehealth/toolkit`
2. Add subdirectory: `:packages/toolkit`
3. Update import paths if using modules
4. See [Migration Guide](./monorepo-migration-guide.md) for full details

**Important:** Only consumers using `#main` branch or no version will be affected immediately. Version tags (v3.4.2, v3.4.1, etc.) continue to work.

### If You're Starting Fresh

Just follow the new installation pattern:

```json
{
  "dependencies": {
    "@ourfuturehealth/toolkit": "github:ourfuturehealth/design-system-toolkit#toolkit-v4.0.0:packages/toolkit",
    "@ourfuturehealth/react-components": "github:ourfuturehealth/design-system-toolkit#react-v0.0.1:packages/react-components"
  }
}
```

## FAQ

### Will this break existing consumers?

**No, not if they're using version tags.**

Projects using specific version tags (e.g., `#v3.4.2`) will continue to work indefinitely because:

- Git tags are immutable
- v3.4.2 will always point to the pre-refactor code structure
- npm/pnpm will install from that exact commit

**Projects that WILL be affected:**

- Using `#main` branch instead of a version tag
- No version specified (defaults to latest/main)
- Trying to upgrade to toolkit-v4.0.0 without updating syntax

**Safe migration path:** Consumers can stay on v3.4.2 as long as needed, then upgrade to v4.0.0 when ready using the new installation syntax.

### Why use `toolkit-v*` for toolkit releases?

To clearly identify package-specific releases in a multi-package repository. Since v4.0.0 introduced the monorepo structure, we use prefixed tags to distinguish between toolkit and React component releases.

### Can I still download the zip bundles?

**Yes!** Toolkit releases (`v*`) include the bundle zip as before:

- Download from GitHub Releases
- Or install via git and access `node_modules/@ourfuturehealth/toolkit/dist/`

### Do both packages always release together?

**No.** Each package has independent versioning and can be released separately:

- Change only toolkit CSS → Release toolkit only (`toolkit-v4.0.1`)
- Change only React components → Release react-components only (`react-v0.0.2`)
- Change both → Release both (create two tags)

### What about the source code zip/tar.gz in releases?

GitHub automatically generates source code archives for every release. These now contain the **entire monorepo**, not just one package.

**For consuming code:**

- ✅ **Use git installation** with subdirectory: `#toolkit-v4.0.0:packages/toolkit`
- ✅ **Download the toolkit bundle zip** for pre-built assets
- ❌ Don't use the auto-generated "Source code" zip unless you want the entire monorepo

- ✅ Use `toolkit-v*` for toolkit (e.g., `toolkit-v4.0.0`)
